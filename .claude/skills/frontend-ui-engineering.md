# Frontend UI Engineering

Build UI that looks like it was built by a design-aware engineer at a top company — not like it was generated by an AI.

## When to Use

- Building new UI components or pages
- Modifying user-facing interfaces
- Implementing responsive layouts
- Adding interactivity or state management
- Fixing visual or UX issues

## Component Architecture

### File Organization
Organize files by feature with colocated concerns:
```
components/
  stock-quote/
    stock-quote.tsx      # Main component
    stock-quote.test.tsx # Tests
    use-stock-data.ts    # Custom hook
    types.ts             # TypeScript types
```

### Design Principles
- **Composition over configuration** - Build small, focused components that compose together
- **Single responsibility** - Each component does one thing well
- **Presentation/container split** - Separate data fetching from rendering
- **Prop drilling limit** - If drilling deeper than 3 levels, consider context or state management

## State Management Hierarchy

Choose the simplest approach that fits the use case:

1. **Local state (useState)** - Component-specific UI state
2. **Lifted state** - Shared state between siblings via parent
3. **Context** - Theme, auth, feature flags
4. **URL state** - Filters, pagination, search (shareable)
5. **Server state (React Query/SWR)** - API data with caching
6. **Global store (Zustand)** - Complex client-side state

## Design System Adherence

### Anti-Patterns to Avoid (AI Aesthetic)
- Purple/indigo as primary color
- Excessive gradients everywhere
- Maximum border-radius on everything
- Generic card grids with equal spacing
- Placeholder "Lorem ipsum" copy
- Oversized uniform padding
- Heavy drop shadows on every element
- Hero sections with stock illustrations

### Do Instead
- Use project's design tokens from DESIGN.md
- Follow spacing scale consistently (4px base)
- Content-first layouts
- Purposeful whitespace
- Real or realistic copy
- Subtle shadows only where needed
- Brand-appropriate colors

## Accessibility (WCAG 2.1 AA)

### Keyboard Navigation
Every interactive element must be keyboard accessible:
```tsx
// Good: Proper button semantics
<button onClick={handleClick}>Submit</button>

// Bad: Non-semantic clickable div
<div onClick={handleClick}>Submit</div>
```

### ARIA Labels
Add labels for unlabeled interactive components:
```tsx
<button aria-label="Close dialog">
  <XIcon />
</button>
```

### Focus Management
```tsx
const inputRef = useRef<HTMLInputElement>(null);

useEffect(() => {
  if (isOpen) {
    inputRef.current?.focus();
  }
}, [isOpen]);
```

### Empty & Error States
Always provide meaningful feedback:
```tsx
{data.length === 0 ? (
  <EmptyState
    icon={<SearchIcon />}
    title="No results found"
    description="Try adjusting your search criteria"
    action={<Button onClick={clearFilters}>Clear filters</Button>}
  />
) : (
  <DataTable data={data} />
)}
```

## Responsive Design

### Mobile-First Approach
```tsx
// Start with mobile, add breakpoints up
<div className="p-4 md:p-6 lg:p-8">
  <div className="grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3 gap-4">
```

### Test Breakpoints
- 320px (small phones)
- 768px (tablets)
- 1024px (laptops)
- 1440px (desktops)

### Touch Targets
Minimum 44x44px for interactive elements on mobile.

## Loading States

### Skeleton Screens (Preferred)
```tsx
{isLoading ? (
  <div className="space-y-2">
    <Skeleton className="h-6 w-32" />
    <Skeleton className="h-4 w-48" />
    <Skeleton className="h-4 w-40" />
  </div>
) : (
  <StockInfo data={data} />
)}
```

### Optimistic Updates
```tsx
const handleLike = async () => {
  // Update UI immediately
  setLiked(true);
  setLikeCount(prev => prev + 1);
  
  try {
    await api.like(postId);
  } catch (error) {
    // Revert on failure
    setLiked(false);
    setLikeCount(prev => prev - 1);
  }
};
```

## Code Quality Checklist

### Red Flags
- [ ] Component > 200 lines (split it)
- [ ] Inline styles (use Tailwind/CSS)
- [ ] Missing loading state
- [ ] Missing error state
- [ ] Missing empty state
- [ ] No keyboard support
- [ ] Hardcoded colors (use tokens)
- [ ] Magic numbers (use spacing scale)

### Verification Steps
1. Renders without errors in dev and prod
2. Passes TypeScript strict mode
3. Keyboard navigable
4. Screen reader announces content correctly
5. Works at all breakpoints
6. Loading states show skeletons
7. Errors display helpful messages
8. Matches design system tokens

## Financial UI Specifics

### Number Formatting
```tsx
// Use tabular numbers for alignment
<span className="font-mono tabular-nums">
  {formatCurrency(price)}
</span>
```

### Stock Price Changes
```tsx
<span className={cn(
  "font-mono font-medium",
  change > 0 && "text-green-600",
  change < 0 && "text-red-600",
  change === 0 && "text-gray-500"
)}>
  {change > 0 ? '+' : ''}{formatPercent(change)}
</span>
```

### Data Tables
- Right-align numeric columns
- Use monospace font for numbers
- Sticky headers for scrollable tables
- Sort indicators on sortable columns
- Row hover states

### Charts
- Use consistent color palette from DESIGN.md
- Always include axis labels
- Provide tooltip on hover
- Handle empty data gracefully
- Responsive sizing
